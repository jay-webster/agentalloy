# automation-integrator-intake — Design

## Approach

### 1. New `automation/integrator.py` module, sibling to `injection_guard.py`

**Decision.** Not folded into `store.py` — same separation-of-concerns
reasoning as slice 3 (`injection_guard.py` is a content-analysis concern
the store calls; `integrator.py` is a file-generation concern the store
also calls, but neither is itself storage logic). Two functions:

```python
def slugify(subject: str, message_id: str) -> str:
    # lowercase, non-alnum -> "-", collapse/strip hyphens, truncate to 40
    # chars, then append "-" + message_id[:8] -- always unique by
    # construction, no collision-detection needed
    ...

def render_draft(candidate: Candidate, slug: str) -> str:
    # returns the draft file's full markdown content, templated from
    # candidate fields -- pure string formatting, no I/O
    ...
```

Splitting `render_draft` out as a pure function (not doing the file write
itself) makes it trivially unit-testable (AC3) without touching the
filesystem, matching the rest of this codebase's preference for pure
functions over methods with side effects baked in.

### 2. Draft format: quote fields plainly, don't imitate a real spec

**Decision (resolves the spec's open design question).** The draft is
explicitly NOT structured with `## Acceptance Criteria` / `## Out of
Scope` headings — using those headings on unfilled content would either
sit there as `<placeholder>` text indistinguishable from a real scaffold's
placeholders, or invite copy-paste-without-thinking. Instead:

```markdown
# Draft intake: {subject}

> Generated by the automation pipeline's integrator from an accepted
> candidate. This is NOT a spec — it's source material for one. Run
> `agentalloy contract init --phase spec --slug {slug}` and write a real
> spec, using the material below as a starting point.

**Source**: {source}
**Received**: {received_at}
**Message ID**: {message_id}

## Why this was accepted

{rationale}

## Original content

{snippet}
```

The header explicitly says "this is NOT a spec" — a deliberate guard
against the draft being mistaken for (or lazily treated as) real spec work,
consistent with the spec's Assumptions.

### 3. Storage: `integrated_at` + `integration_slug`, same migration pattern

**Decision.** Extend `_NEW_COLUMNS` (the mechanism slices 2-3 already
built) with `integrated_at: "TEXT"` and `integration_slug: "TEXT"`. New
`CandidateStore.integrate(message_id: str) -> IntegrationResult` where
`IntegrationResult` is a small frozen dataclass (`slug: str, draft_path:
Path, already_existed: bool`) — richer than a bare `bool` because the CLI
needs to report the path either way (new draft or pre-existing one).

Logic:
1. Look up the row; missing → raise (mirrors `evaluate()`'s pattern of
   raising rather than returning a sentinel for "doesn't exist" — chosen
   for consistency within this module rather than re-litigating `mark`'s
   older bool-return convention).
2. `verdict != "accept"` → raise `NotAcceptedError`.
3. `integrated_at is not None` → return `IntegrationResult(slug=<existing
   integration_slug>, draft_path=<computed from it>, already_existed=True)`
   — **no file write, no row update**. This is the idempotency guarantee
   (AC2): the check happens before any I/O, not after.
4. Otherwise: `slugify()`, `render_draft()`, write the file (creating
   `automation/intake-drafts/` if needed), `UPDATE` the row, return
   `IntegrationResult(..., already_existed=False)`.

### 4. CLI: `ingest integrate <message_id>`

**Decision.** Wraps `store.integrate()`; catches `NotAcceptedError` (clear
message naming the actual verdict, exit 1) and the missing-candidate case
(same shape as existing commands); on success prints the draft path and,
when `already_existed` is true, says so explicitly ("already integrated,
draft unchanged") rather than implying fresh work happened.

## Non-goals carried from spec

No SDD lifecycle execution. No PR opening. No notification wiring. No
re-verdicting real candidates for live-proof purposes — synthetic test data
only, cleaned up after.
